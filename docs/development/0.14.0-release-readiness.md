# SwiftFTR 0.14.0 Release Readiness

This document is the release gate for SwiftFTR 0.14.0. The release must be tagged from the
release-preparation PR's merge commit on `main`, not from the branch commit.

## Status

The release-preparation branch contains the adopter-facing notes, migration guide, version and
dependency updates, compatibility overloads, and release-workflow maintenance. Do not create the
`v0.14.0` tag until every pre-tag item below is complete on the final mainline commit.

## Live acceptance matrix

Discover interface names and addresses from the operating system for every run. Never infer a
physical or VPN role from a BSD-name prefix or numeric suffix.

- [x] Direct dual-stack network (validated 2026-07-21 using OS-discovered routes and addresses):
  - default-route IPv4 and IPv6 ping;
  - bound IPv4 and IPv6 ping using the exact selected interface and source address;
  - IPv4 and IPv6 traceroute plus streaming trace;
  - IPv4 and IPv6 TCP probes;
  - bound IPv4 and IPv6 UDP loopback probes;
  - A and AAAA record requests over both IPv4 and IPv6 DNS transports;
  - dual-stack STUN public-address discovery; and
  - embedded IPv4 and IPv6 ASN lookup.
- [ ] VPN connected:
  - record OS-reported default routes and active interfaces;
  - repeat default-route IPv4/IPv6 checks with outcomes interpreted against the VPN policy;
  - verify caller-selected exact-interface probes either use the selected route or fail explicitly;
  - never accept silent fallback to a different interface.
- [ ] VPN disconnected again:
  - confirm default routes and public addresses return to the direct-network baseline;
  - confirm the former provider interface disappears or exact binding to it fails; and
  - repeat the direct IPv4/IPv6 smoke set.

The exact-interface HTTPS work in PR #53 already completed an off/on/off ProtonVPN rehearsal with
packet-capture evidence. Repeat only a concise smoke on the final release SHA unless the internal
spike or its Network.framework dependencies change.

The checked deterministic gates above were rerun on the release-preparation working tree after the
final code and documentation edits. The offline suite completed with 287 Swift Testing cases plus
the XCTest suites and no failures. Debug and release builds were warning-free. Repeat the concise
checks required above after the PR is merged because the merge commit is the tag candidate.

## Tag and publication sequence

1. Create an annotated `v0.14.0` tag on the release-preparation merge commit.
2. Push only that tag and watch `Release (attested + SBOM)` complete.
3. Verify the ARM64 CLI, SHA-256 file, single `sbom.cdx.json`, and provenance attestation.
4. Replace auto-generated GitHub notes with the curated 0.14.0 changelog section and add the
   `v0.13.0...v0.14.0` comparison link.
5. Confirm the checksum matches the published binary.
6. Build a throwaway external Swift package using the remote repository with `exact: "0.14.0"`.
7. Update NWX to the published tag, refresh package resolution, and run its SwiftFTR integration
   tests before broad rollout.

## Explicitly not blocking 0.14.0

- The exact-interface `NWConnection` HTTPS implementation remains an internal experiment. It is not
  wired into public `httpProbe`, so the production HTTP Probe roadmap is follow-up work rather than
  a 0.14.0 release gate.
- DNS-over-TCP fallback for truncated UDP replies remains unimplemented and is documented in the
  changelog and migration guide.
- Multipath remains intentionally IPv4-only.
- The self-hosted `local-integration-tests` workflow still uses its older checkout action until the
  dormant runner is upgraded and validated for the Node 24 Actions runtime. Local execution of the
  live matrix is the 0.14.0 gate.
- Historical release notes retain literal interface names where they record past observations.
  Current examples, discovery, fixtures, and runtime logic do not assign roles from those names.
